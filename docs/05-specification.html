1 システム概要

システムの目的

解決する課題
- 予約情報とメール会員情報が分散管理されると、集計・分析や施策判断が遅れる。

提供する価値
- 空き状況照会から予約確定までを自社サイトで完結し、受付業務を効率化する。
- 定員チェックと予約番号発番を自動化し、受付品質と運用の一貫性を高める。
- 管理APIで予約・会員データを可視化し、日次/週次の運営判断を迅速化する。
- メール登録/解除を一元管理し、継続的な顧客接点づくりを支援する。

システム構成図

[利用者ブラウザ]
- 静的画面表示（index/workshop/registration など）
- 予約・メール登録のAPI呼び出し（fetch）

▼

5 非機能要件

セキュリティ要件

1. 認証・認可
- 管理機能（/admin.html, /api/admin/**）は管理者ロール（ROLE_ADMIN）でのみアクセス可能。
- ログインはフォーム認証（/login）を使用し、失敗時はエラー画面へ遷移。
- 公開API（予約・メール登録系）は匿名アクセスを許可。

**2. 認証情報管理
- 管理者パスワードはハッシュ化（BCrypt）して照合する。
- 管理者ID/パスワードは環境変数または設定ファイルで外部化し、デフォルト値運用を禁止する。
- 本番環境での認証情報ローテーション手順を定義する。

3. 入力値検証
- 予約・メール登録APIは Bean Validation により必須・形式・範囲を検証する。
- 業務バリデーション（予約日範囲、時間帯、定員超過、コース値）をサーバー側で強制する。
- 不正入力時は適切なHTTPステータス（400/409）を返却する。

4. 通信保護
- 本番はHTTPS必須（TLS1.2+）。
- セッションCookieは Secure / HttpOnly / SameSite を有効化する。

5. CSRF対策
- 現状はCSRF無効化のため、要件としては以下のいずれかを必須化する。
　1. CSRFトークンを有効化（推奨）
　2. 同一オリジン制約と厳格なCORS、かつ状態変更APIに追加対策を実装

6. CORS制御
- 現状 * 許可のため、本番では許可オリジンを明示的に限定する。
- 許可メソッド・許可ヘッダーを最小化する。

7. 監査・ログ
- ログイン成功/失敗、管理APIアクセス、予約登録、会員登録/解除を監査ログに記録する。
- 個人情報（メール、電話番号等）はマスキング方針を設ける。

8. 可用性・不正利用対策
- ログイン試行回数制限（ブルートフォース対策）を設ける。
- 予約/登録APIにレート制限を導入する。
- 例外時にスタックトレースや内部情報をクライアントへ返さない。

9. データ保護
- 個人情報の保存期間・削除ルールを定義する。
- バックアップとリストア手順を整備し、定期検証する。

対応環境

クライアント環境（利用者側）
- 対応デバイス: PC / タブレット / スマートフォン（レスポンシブ）
- ブレークポイント: 1280px（PC⇔SPナビ切替）、900px（モバイル最適化）
- 対応ブラウザ: fetch / CSS Grid / matchMedia が使えるモダンブラウザ
（Chrome / Edge / Safari / Firefox の最新安定版を推奨）
- JavaScript: 有効必須（予約・管理画面のAPI連携で使用）

サーバー環境（アプリ実行）
- 言語・ランタイム: Java 17
- フレームワーク: Spring Boot 3.4.4
- ビルド: Maven（mvnw/mvnw.cmd 同梱）
- 実行形態: 組み込みサーバー（Spring Boot標準）

データベース環境
- DB: H2 Database（ファイルDB）
- 接続: jdbc:h2:file:./data/reservationdb
- 補助機能: H2 Console（/h2-console）

開発環境（推奨）
- OS: Windows / macOS / Linux（Java 17 が動作する環境）
- 必須ツール: JDK 17、Maven（または Maven Wrapper）
- 文字コード: UTF-8推奨
